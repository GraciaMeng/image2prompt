# Image to Prompt 真机冒烟与发布执行单

适用对象：测试、产品、运营、发布负责人

用途：用于 `Image to Prompt` 在提测、灰度、正式售卖上线、提交 Chrome Web Store 前的统一执行。本文档按“先自动校验，再真机冒烟，最后商店提交检查”的顺序编写，可直接照单执行并留档。

## 1. 本次执行目标

- 确认 Chrome 扩展生产包可正常加载。
- 确认网页图片上的 hover 入口和设置跳转可用。
- 确认图片分析链路支持流式输出、复制结果、失败后重试。
- 确认成功结果会写入本地记录，设置页可看到真实记录而非仅示例卡片。
- 确认不同站点、不同图片格式下的基础兼容性。
- 确认商店提交前的文案、素材、隐私、权限和产物检查已齐备。

## 2. 执行前准备

### 2.1 环境

- 操作系统：macOS 或 Windows 任一真机环境。
- 浏览器：Chrome 最新稳定版。
- 浏览器配置：建议单独新建一个全新 Chrome Profile，避免历史扩展、缓存、登录态干扰。
- 扩展产物：项目根目录 `dist/`。
- 模型服务：一套真实可用的 OpenAI-compatible 多模态接口。

### 2.2 必备资料

- 测试 API Key
- 可访问的 `Base URL`
- 可处理图片并支持 `stream: true` 的模型名
- 用于留档的截图目录
- 本地记录检查方式：Chrome 扩展设置页中的“最近分析记录”

### 2.3 测试素材

- `JPG/JPEG` 图片 1 张
- `PNG` 图片 1 张
- `WEBP` 图片 1 张
- `GIF` 图片 1 张
- 超大图片 1 张
- `SVG` 图片 1 张
- 至少 3 类站点页面
  - 普通内容站/文章页
  - 电商详情页
  - 懒加载图片流页面

建议优先使用可公开展示、版权可控的测试页面和图片。

## 3. 发布前自动校验

执行人：研发或发布负责人

执行命令：

```bash
npm run check
```

通过标准：

- `typecheck` 通过
- `test:run` 通过
- `build` 通过并生成 `dist/`
- `verify:dist` 通过

重点核对：

- `dist/manifest.json` 存在
- 生产包不包含 `alarms`
- 生产包不保留 `hot-update.json` 相关声明
- `background.js`、`content.js`、`index.html`、图标产物齐全

失败处理：

- 任一步失败，本轮不进入真机冒烟，先退回研发修复。

证据留存：

## 5. 站点兼容专项检查

执行人：测试

目的：验证“不是只在单一测试站点可用”，避免正式售卖后出现大面积站点不兼容。

| 站点类型 | 建议检查点 | 通过标准 | 备注记录方式 |
| --- | --- | --- | --- |
| 普通文章页 | hover 能否稳定识别正文图片 | 能出现 hover 入口并成功分析 | 记录站点域名、图片格式、结果 |
| 电商详情页 | 主图、缩略图、轮播切换后图片是否可触发 | 至少主图可分析；如缩略图不可触发需说明原因 | 记录是主图失败还是缩略图失败 |
| 懒加载流页面 | 首屏外图片滚动进入视口后是否可触发 | 图片加载完成后 hover 入口出现，分析链路可启动 | 记录是否需等待图片完全加载 |

专项结论标准：

- 至少 2 类站点成功。
- 若有站点失败，必须记录失败域名、失败现象、是否为站点防盗链/CSP/图片地址不可拉取导致。
- 兼容性问题不允许写成“偶现”，必须给出具体页面和复现方式。

## 9. 放行建议

- `P0` 任一失败，不允许提交商店，不允许正式售卖上线。
- `P1` 如有失败，可视影响范围决定是否带风险放行，但必须在发布单写清楚受影响站点、图片格式或交互路径。
- 若“记录落库”失败，建议视为高优先级问题，因为这会直接影响产品的“可复看、可沉淀”售卖价值。
